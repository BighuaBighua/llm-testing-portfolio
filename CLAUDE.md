# 项目开发规范

## 禁止行为
未经"执行"批准不得创建/修改/删除文件或运行命令；未理解代码前不得提方案；不得跳过 brainstorming；不得无证据声称完成。

## 批准规则
用户说"执行"前仅为讨论，不得行动。

## 理解优先
先完全理解项目再提方案。探索所有相关文件；给完整方案不分多次；方案批准后再建任务。

**特别强调**：
- 每次调用 Skill、给出建议、执行修改前，必须重新读取 CLAUDE.md 确认最新版本
- 禁止基于记忆中的旧版本内容做决策
- 每次行动前必须验证："我是否刚刚读取了最新文件？"

## 强制行为规则

**当用户输入包含以下任一触发词时，你必须立即（在该消息的第一时间）调用 Skill 工具，不得继续正常回复：**

### brainstorming 触发条件

| 你的口语表达 | 核心意图 | 调用方式 |
|------------|---------|---------|
| "清洗这个文件"、"清理一下"、"整理一下"、"优化这个项目"、"精简" | 清理/优化类 | `Skill(name="brainstorming")` |
| "哪些...需要/应该/可以..."、"是不是应该..."、"哪些文件废弃了"、"哪些目录需要提交" | 判断类 | `Skill(name="brainstorming")` |
| "给个方案"、"分析一下"、"思考一下"、"怎么弄"、"怎么做"、"你看一下...给个方案" | 方案设计类 | `Skill(name="brainstorming")` |
| "最终目的是..."、"我想要..."、"我的目的是让..."、"希望..."、"核心目的是..." | 目标明确类 | `Skill(name="brainstorming")` |
| "能不能..."、"可不可以..."（涉及结构性修改） | 询问可行性 | `Skill(name="brainstorming")` |

### systematic-debugging 触发条件

| 你的口语表达 | 核心意图 | 调用方式 |
|------------|---------|---------|
| "Bug"、"错误"、"失败"、"不行"、"有问题"、"为什么不工作" | 报告问题 | `Skill(name="systematic-debugging")` |
| "为什么没调用"、"为什么没有执行" | 排查原因 | `Skill(name="systematic-debugging")` |

### writing-plans 触发条件

| 你的口语表达 | 核心意图 | 调用方式 |
|------------|---------|---------|
| "开始实现"、"写代码"、"执行计划"、"开始做"、"执行" | 开始实施 | `Skill(name="writing-plans")` |

### verification-before-completion 触发条件

| 你的口语表达 | 核心意图 | 调用方式 |
|------------|---------|---------|
| "完成了"、"做完了"、"修好了"、"搞定" | 声称完成 | `Skill(name="verification-before-completion")` |

### 无需调用技能的情况

- **读取/解释**："看一下"、"读一下"、"解释一下"、"什么意思"
- **纯询问**："为什么"（非排查 Bug）、"你的意思是..."、"确认一下"
- **示例**：
  - "你看一下这个 tree" → 无需技能
  - "为什么没有调用呢"（询问原因）→ 无需技能
  - "你的意思是在 project rules 规定了..." → 无需技能

## 技能连贯使用规则

**标准工作流**（新功能/重构/清理/优化）：

```
brainstorming → writing-plans → test-driven-development → verification-before-completion
```

**Bug 修复工作流**：

```
systematic-debugging → test-driven-development → verification-before-completion
```

**强制规则**：
1. ✅ 不得跳过任何阶段
2. ✅ 每个阶段完成后必须获得用户批准（用户说"执行"、"同意"、"确认"、"开始"）
3. ✅ 声称完成前必须提供验证证据（测试通过日志/命令输出）
4. ✅ 每次回复时必须说明当前阶段状态

## 错误示范 vs 正确做法

**❌ 错误示范（当前问题）**：

```
用户："清洗这个文件"
AI：识别到触发词 → 继续正常回复 → 后续才想调用技能
```

**✅ 正确做法**：

```
用户："清洗这个文件"
AI：识别到触发词 → 第一反应调用 Skill(name="brainstorming") → 停止其他回复 → 等待技能执行
```

## 系统性思维
主动检查相关问题，一次性解决，全面验证。

## 文档规范

### 设计文档存放路径

**所有设计文档必须存放在以下目录**：
```
archive/superpowers/specs/YYYY-MM-DD-<topic>-design.md
```

**完整路径示例**：
```
/Users/honey/Desktop/llm-testing-portfolio/archive/superpowers/specs/2026-04-12-deprecated-files-cleanup-design.md
```

**适用范围**：
- brainstorming 技能生成的设计文档
- writing-plans 技能生成的实现计划
- 所有架构设计、技术方案、重构计划等文档

**规则说明**：
- 日期格式：YYYY-MM-DD（如 2026-04-12）
- 主题命名：kebab-case（如 deprecated-files-cleanup）
- 文件后缀：统一使用 `-design.md`
- 目录不存在时自动创建

**历史原因**：
- 原默认路径：`docs/superpowers/specs/`
- 现统一路径：`archive/superpowers/specs/`
- 目的：集中管理所有历史文档和设计方案
