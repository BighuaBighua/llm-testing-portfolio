# 性能优化建议

> 基于 LLM 评测系统实际架构的性能优化策略

## 🎯 性能优化概述

### 性能瓶颈分析

LLM 评测系统的性能瓶颈主要集中在 API 调用环节：

| 瓶颈 | 原因 | 影响 |
|------|------|------|
| API 调用延迟 | LLM 推理耗时 | 单条用例 3-10 秒 |
| API 限流 | Provider 速率限制 | 429 错误导致重试 |
| 串行执行 | 默认单线程模式 | 总耗时 = 用例数 × 单条耗时 |
| 网络波动 | 不稳定连接 | 超时和重试 |

### 优化目标

- **减少总评测时间**：从串行到并发
- **提高 API 可用性**：Fallback 机制
- **降低重试成本**：合理超时和重试策略
- **优化资源使用**：缓存和批量处理

## 🔧 配置层优化

### 1. 执行模式选择

`execution.yaml` 提供三种并发模式：

| 模式 | max_concurrent | delay | 适用场景 |
|------|---------------|-------|----------|
| `single_thread` | 1 | 2.0s | 调试、API 限流严格 |
| `concurrent` | 2 | 0.5s | 日常评测（推荐） |
| `performance` | 5 | 0.1s | 大批量评测、API 充足 |

```yaml
concurrency:
  default_mode: "concurrent"
```

### 2. 超时与重试参数

```yaml
parameters:
  timing:
    api_timeout: 60
    case_timeout: 300
  retry:
    max_attempts: 3
    backoff_factor: 2
    max_delay: 60
```

**优化建议**：
- `api_timeout`：简单问答设 30s，复杂评测设 60-90s
- `case_timeout`：单轮 120s，多轮 300s
- `max_attempts`：3 次足够，过多浪费时间
- `backoff_factor`：指数退避避免限流加剧

### 3. 推理参数优化

```yaml
parameters:
  inference:
    under_test:
      temperature: 0.7
      top_p: 0.9
    evaluator:
      temperature: 0.3
      top_p: 0.9
```

**优化建议**：
- 被测模型 `temperature: 0.7`：保持回答多样性
- 评测模型 `temperature: 0.3`：降低判定随机性
- 评测模型低温度确保判定一致性

### 4. 质量门禁阈值

```yaml
quality_gate:
  overall_threshold: 0.9
  critical_dimensions:
    - "compliance"
    - "security"
  critical_threshold: 0.95
```

## 🚀 API 层优化

### 1. Fallback 机制

在 `api_config.yaml` 中配置备用 Provider：

```yaml
evaluator:
  name: "阿里云DashScope"
  model: "qwen-turbo"
  fallback:
    enable: true
    providers:
      - name: "魔搭社区"
        model: "Qwen/Qwen3.5-35B-A3B"
        priority: 2
```

**优化效果**：
- 主 Provider 不可用时自动切换
- 避免单点故障导致评测中断
- 不同 Provider 可并行尝试

### 2. API 调用间隔

`generation_settings` 中的 `api_delay` 控制调用间隔：

```yaml
generation_settings:
  batch_size: 5
  api_delay: 2.0
  max_retries: 2
```

**优化建议**：
- 千帆：`api_delay: 1.0`（限流较宽松）
- DashScope：`api_delay: 2.0`（默认推荐）
- 魔搭：`api_delay: 3.0`（限流较严格）

### 3. 批量生成策略

`batch_size` 控制每次 API 调用生成的用例数：

- `batch_size: 5`：平衡速度和质量
- `batch_size: 10`：更快但可能降低单条质量
- `batch_size: 3`：更高质量但更慢

## 💾 缓存层优化

### 1. PromptTemplateLoader 缓存

模板加载器使用实例级缓存，避免重复 IO：

```python
loader = PromptTemplateLoader()
loader.render("generation/standard.md", variables)  # 首次加载
loader.render("generation/standard.md", variables)  # 命中缓存
```

**优化建议**：
- 在同一评测批次中复用 `PromptTemplateLoader` 实例
- 避免频繁创建新实例导致缓存失效

### 2. ConfigLoader 缓存

配置加载器使用实例级缓存，每个配置文件只加载一次：

```python
loader = ConfigLoader()
loader.load_business_rules()  # 首次加载
loader.load_business_rules()  # 命中缓存
```

### 3. ConfigRegistry 单例

全局单例避免重复初始化：

```python
registry = ConfigRegistry.get_instance()  # 复用全局实例
```

## 📊 数据层优化

### 1. 用例数量优化

根据实际需求调整各维度用例数量：

| 维度 | 默认数量 | 优化建议 |
|------|---------|---------|
| 标准维度（accuracy等） | 10 | 快速验证可减至 5 |
| prompt_injection | 10 | 安全评估不建议减少 |
| sensitive_topic | 30 | 可按话题类型选择性减少 |
| bias_fairness | 30 | 可按偏见类型选择性减少 |
| multi_turn | 10 | 场景覆盖不建议减少 |

### 2. CSV 导出优化

`csv_export_config` 控制导出字段：

```yaml
csv_export_config:
  encoding: "utf-8-sig"
```

使用 `utf-8-sig` 编码确保 Excel 正确显示中文。

### 3. Bad Case 去重

`BadCaseManager._deduplicate()` 自动去重，避免重复分析：

- 同一 `source_test_case_id` 的 Bad Case 自动合并
- `occurrence_count` 累加追踪频率
- 严重程度自动升级

## 🔄 流程层优化

### 1. 增量评测

使用 `--dimensions` 参数只评测特定维度：

```bash
python3 scripts/run_tests.py --dimensions compliance,prompt_injection
```

### 2. 追加生成

使用 `--append` 模式追加用例而非全量重新生成：

```bash
python3 scripts/generate_test_cases.py --dimensions sensitive_topic --append
```

### 3. 中断恢复

利用 `TestRunRecorder.detect_interruption()` 从断点恢复，避免重复执行。

## 📈 性能基准参考

### 典型评测耗时

| 用例数 | 模式 | 预估耗时 |
|--------|------|---------|
| 50 | single_thread | 15-25 分钟 |
| 100 | single_thread | 30-50 分钟 |
| 100 | concurrent | 15-25 分钟 |
| 100 | performance | 8-15 分钟 |
| 200 | performance | 15-30 分钟 |

### API 调用成本估算

| 维度 | 用例数 | API调用次数 | 预估Token |
|------|--------|------------|----------|
| 标准维度×9 | 90 | 90×2=180 | ~180K |
| prompt_injection | 10 | 10×2=20 | ~20K |
| sensitive_topic | 29 | 29×2=58 | ~58K |
| bias_fairness | 30 | 30×2=60 | ~60K |
| **合计** | **159** | **318** | **~318K** |

> 每条用例需调用被测模型和评测模型各一次，故 API 调用次数 = 用例数 × 2
