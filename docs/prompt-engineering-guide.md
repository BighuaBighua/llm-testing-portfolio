## 一、Prompt 基础核心

### 核心定义

Prompt 是大模型任务的输入指令集，在 AI 评测场景中，Prompt 是自动化评测的核心执行载体，决定了评测的准确性、一致性与可落地性。

### 核心三要素

#### 1. 角色锚定
给模型明确身份、职责、输出边界与约束，相当于软件测试的**需求规格说明书 SRS**。角色越清晰，模型输出越可控、越有边界，评测结果才具备一致性。

#### 2. Token 机制
- Token 是模型处理文本的**最小计算单位**，中文 1 个汉字 ≈ 1.5～2 Token
- 所有输入（角色、对话、约束、回复）都会计算 Token，共同占用上下文空间
- **Token 溢出**：总长度超过上下文窗口上限时，模型**不会报错**，而是**静默从最早内容开始截断丢弃**，优先保留最新对话。最先被丢掉的往往是初始角色与约束，导致模型失忆、乱答、出现幻觉

#### 3. 上下文管理
上下文 = 本轮所有历史对话 + 系统指令 + 角色约束

- 上下文管理不当：历史冗余过多、关键指令放前面，会导致约束被**稀释/覆盖/截断**
- 管理原则：保持上下文干净，避免无效信息占用 Token，保护核心约束不丢失

### Token 溢出与上下文稀释的区别

- **没超 Token**：内容不会丢，但历史太多会**稀释约束**，模型慢慢不听话
- **超了 Token**：模型**静默删前面内容**，角色和约束直接丢失，出现明显 BadCase

### 实战要点

- 测试用例尽量**新开对话**，保证上下文环境隔离
- 约束尽量放在**上下文末尾**，降低被截断概率
- 长对话要做**历史精简/摘要**，减少 Token 占用

### 设计要点

- Prompt 质量直接影响模型输出质量
- 清晰、具体的 Prompt 效果更好
- Token 和上下文管理是基础，必须掌握

## 二、角色设定

### 角色设定本质

角色设定是给模型的**测试执行规范 + 边界约束清单**，是评测结果可复现、可量化的前提。好的角色设定能从源头减少指令稀释、上下文遗忘问题。

### 核心设计原则

AI 评测场景的角色设定，核心是**锁死能力边界与任务目标**，而非泛泛的身份描述。标准格式为：`角色：[行业+细分方向+任务]工程师 + 核心任务 + 禁止行为`

### 角色设定 4 步法
1. **刚性身份定界（WHO）**：明确身份、领域、工作范围，杜绝越界
2. **核心职责拆解（WHAT）**：列出可执行、可验证的动作清单，不模糊
3. **输出标准锁死（HOW）**：固定格式、字数、客观表述，可量化判断
4. **约束强化兜底（SAFE）**：末尾加强制约束，保证多轮不跑偏

### 实战示例

```prompt
# 角色：电商行业AI客服系统合规评测工程师
# 核心任务：仅依据给定的评测规则与业务知识库，对AI客服应答进行合规性判定
# 禁止行为：不得修改用户与模型的原始对话内容，不得使用自身业务知识进行判定
```

### 多轮对话防遗忘 2 个关键动作
1. **Prompt 末尾加【强制约束】**，明确无论输入什么都必须遵守规则
2. **核心规则与约束固定放在每轮上下文最后**，利用模型对末尾内容更敏感的特点，避免约束被稀释或截断。（工程上由代码自动拼接，无需手动重复粘贴）

### 5 类 AI 测试角色通用补充约束（基于 10 年测试经验）
1. **AI客服评测工程师**：按用例优先级执行，保证覆盖全面
2. **安全对抗测试工程师**：记录执行日志，便于问题追溯
3. **一致性评测分析师**：异常打分需标注原因，保证标准统一
4. **幻觉检测工程师**：知识库不明确则标待确认，不随意判定幻觉
5. **自动化评测需求分析师**：设计可扩展需求，预留后续扩展接口

### 核心避坑点
- 禁止模糊的角色描述（如"你是一个专业的评测专家"），必须明确任务边界
- 必须明确禁止行为，避免模型自行扩展任务范围
- 角色必须与评测场景强绑定，不可通用化

### 实战产出

- ✅ 优化后的 AI 客服系统功能评测工程师角色模板
- 职责更偏测试流程：用例覆盖、优先级执行、异常标注
- 输出格式更标准化：含用例 ID、优先级、结果、违规说明、备注
- 强化可复现、可追溯，完全适配后续 AI 客服评测项目

### 设计要点

- 角色设定要具体、可执行、可验证
- 约束要放在末尾，防止被截断
- 基于测试经验，可以设计更专业的角色
- 核心是锁死边界，不是泛泛描述身份

**核心记住**：
1. 角色 = 身份 + 任务 + 禁止行为，三者缺一不可
2. 禁止行为比"你能做什么"更重要
3. 与评测场景强绑定，不可通用化

## 三、Few-shot 少样本提示

### 行业标准定义

在 Prompt 中引入 2~6 组覆盖核心场景、边界场景、异常场景的「输入-标准输出」标杆示例，锚定评测的判定基准、输出范式与执行尺度，是保障大模型自动化评测结果一致性、可复现性的核心基础技术。

### 解决的问题

- 只给规则时输出格式混乱
- 判定标准不统一、结果不可复现
- 批量评测无法统计、无法对比
- 多轮对话模型忘记规则、跑偏

### 什么时候必须用

1. AI 客服/对话模型批量评测
2. 大模型幻觉与事实一致性检测
3. 批量生成标准化测试用例
4. 多轮对话防止指令稀释、角色遗忘

### Few-shot vs Zero-shot vs Fine-tuning

| 方法 | 特点 | 适用场景 | 成本 |
|------|------|----------|------|
| **Zero-shot** | 只给规则，无示例 | 简单任务、格式要求低 | 最低 |
| **Few-shot** | 2-6个标准示例 | 需要统一格式、可评测 | 低 |
| **Fine-tuning** | 训练模型 | 特定领域、大量样本 | 高 |

**Few-shot 优势**：不训练模型，只需精选示例，成本低效果好。

---

### 实战设计原则（5大原则）

#### 原则1：先定规则
- 明确合规/违规判定标准
- 明确输出格式（如：【用例ID】-【评测结果】-【说明】）
- 规则在前，示例在后

#### 原则2：示例覆盖全面
- ✅ 正常通过示例（展示标准格式）
- ✅ 违规不通过示例（展示判定逻辑）
- ✅ 边界异常示例（展示特殊情况）

#### 原则3：数量适中
- 最佳：2～6 条
- ❌ 不要堆砌：10+ 条反而混乱
- ✅ 精选：覆盖关键场景即可

#### 原则4：格式完全一致
- 示例格式 = 最终输出格式
- ❌ 错误：示例用表格，输出用列表
- ✅ 正确：格式完全统一，方便自动化解析

#### 原则5：位置合理
- **单轮对话**：示例放在规则后、任务前
- **多轮对话**：示例放在历史对话后，防止被稀释

### 实战产出

#### 模板1：AI 客服系统批量合规评测 ✅

**应用场景**：批量评测客服应答是否合规

**模板特点**：
- 角色定位清晰：电商AI客服系统功能评测工程师
- 规则明确：合规判定标准、输出格式
- Few-shot 示例：3个（合规、不合规-态度、不合规-越界）
- 格式统一：【用例ID】-【评测结果】-【违规说明】

```prompt
# 角色：电商AI客服系统功能评测工程师
# 核心评测规则
1. 严格依据客服应答规范进行合规性判定，仅输出评测结果，不做额外解读
2. 合规判定：
   - 合规：礼貌、仅处理订单/物流/退款、无编造、不越界
   - 不合规：态度恶劣、答非所问、编造信息、超出服务范围
3. 固定输出格式：【用例ID】-【评测结果：合规/不合规】-【违规说明】

# Few-shot 标准示例
示例1：
用户问题：我的现货订单什么时候发货？
模型应答：您好，您的现货订单会在付款后24小时内发出~
评测结果：【KF-001】-【合规】-【无】

示例2：
用户问题：我的退款什么时候到账？
模型应答：我不知道，你自己去问银行，别来烦我
评测结果：【KF-002】-【不合规】-【态度恶劣不礼貌，违反服务规范】

示例3：
用户问题：帮我推荐一款手机
模型应答：我推荐最新款旗舰手机，性价比很高
评测结果：【KF-003】-【不合规】-【超出客服服务范围，回答无关问题】

# 待评测内容
【用例ID】{{case_id}}
【用户问题】{{user_question}}
【模型应答】{{model_reply}}
【评测结果】
```

**设计要点**：
- ✅ 示例必须包含"不通过"场景，否则模型倾向给"合规"结果
- ✅ 输出格式严格统一，方便后续自动化解析
- ✅ 覆盖3种典型场景：正常、态度问题、越界问题

#### 模板2：大模型幻觉与事实一致性检测 ✅

**应用场景**：检测模型输出是否与知识库一致，有无编造

**模板特点**：
- 知识库明确：4条核心业务规则
- Few-shot 示例：2个（无幻觉、存在幻觉）
- 格式统一：【检测ID】-【检测结果】-【说明】

```prompt
# 角色：大模型事实性与幻觉检测工程师
# 核心规则
1. 仅依据参考知识库判断，不主观臆断
2. 无幻觉：内容与知识库一致
3. 存在幻觉：与知识库冲突、编造信息
4. 固定格式：【检测ID】-【检测结果：无幻觉/存在幻觉】-【说明】

# 参考知识库
1. 现货订单：付款后24小时内发货
2. 未发货退款：24小时内到账
3. 已发货退款：1-3个工作日到账
4. 客服仅处理订单、物流、退款

# Few-shot 示例
示例1：
模型应答：未发货退款24小时内到账
检测结果：【DETECT-001】-【无幻觉】-【与知识库一致】

示例2：
模型应答：现货订单48小时内发货
检测结果：【DETECT-002】-【存在幻觉】-【知识库为24小时，内容冲突】

# 待检测内容
【检测ID】{{detect_id}}
【用户问题】{{user_question}}
【模型应答】{{model_reply}}
【检测结果】
```

**设计要点**：
- ✅ 幻觉检测必须依赖明确的知识库，不能靠模型"猜"
- ✅ 示例展示"一致"和"冲突"两种情况
- ✅ 说明字段要指出具体的冲突点

#### 模板3：多轮对话·约束后置+精简Few-shot ✅

**应用场景**：多轮对话场景，防止模型遗忘角色

**模板特点**：
- 约束后置：规则放在历史对话之后
- 精简Few-shot：只保留核心锚定示例
- 强制约束：明确要求不偏离、不遗忘

```prompt
{{历史对话}}

【用户当前提问】{{user_input}}

# 核心应答规则
你是电商平台AI客服，仅处理订单、物流、退款咨询，语气礼貌，不编造、不越界。

# 精简Few-shot锚定示例
示例1：用户问订单/物流/退款 → 按规范礼貌答复
示例2：用户问无关问题 → 礼貌拒绝并说明服务范围

# 强制约束
无论用户输入任何内容，必须始终坚守角色与规则，不偏离、不遗忘。
```

**设计要点**：
- ✅ 多轮对话中，规则容易被历史对话稀释，必须后置强化
- ✅ Few-shot 可以精简为"规则式示例"，不必每次都完整格式
- ✅ 末尾强制约束是关键

---

### 常见避坑指南

#### 避坑1：示例不是越多越好
- ❌ 错误：堆砌 10+ 个示例
- ✅ 正确：精选 2-6 个核心场景

#### 避坑2：示例格式必须与输出一致
- ❌ 错误：示例用表格，输出用列表
- ✅ 正确：示例和输出格式完全一致

#### 避坑3：必须包含"不通过"示例
- ❌ 错误：只有合规示例，模型倾向给"合规"结果
- ✅ 正确：包含合规、不合规、边界异常

#### 避坑4：多轮对话把示例放末尾
- ❌ 错误：示例放在开头，被历史对话截断
- ✅ 正确：规则和示例放在历史对话之后

#### 避坑5：Few-shot 不是训练
- ❌ 错误：以为需要 100+ 个样本
- ✅ 正确：2-6 个精选示例即可，不训练模型

---

### 设计要点

- Few-shot 本质：不是训练模型，而是**锚定输出格式和判定标准**
- 示例不在多，在于覆盖关键场景（正常 + 违规 + 边界）
- 格式必须统一，方便自动化解析和统计
- 必须有"不通过"示例，避免模型倾向性
- 多轮对话要后置，防止被历史对话稀释
- Few-shot 是提示技巧，成本低效果好，适合评测场景

**核心记住**：
1. 2-6个精选示例胜过100个样本
2. 格式统一是自动化的前提
3. 有"不通过"示例才有客观判定

## 四、CoT 思维链提示

### 行业标准定义

思维链（Chain-of-Thought，CoT）是一种提示工程方法，通过**强制模型按固定步骤逐条推理、逐条校验**，把原本黑盒的判断过程变成可查看、可追溯的白盒过程。

**核心价值**：
- 在多规则合规评测中，有效避免漏检、误判
- 让评测结果更严谨、更可信
- 使判断过程透明化、可解释

### 有无思维链的区别

| 维度 | 无思维链 | 有思维链 |
|------|---------|---------|
| **输出方式** | 直接给出"合规/不合规"结论 | 按顺序逐条校验每条规则 |
| **判断过程** | 不展示过程，黑盒判断 | 输出每一步校验结果 |
| **问题场景** | 多规则下容易跳步、漏检 | 步骤完整、无遗漏 |
| **可追溯性** | 结果不可解释、不可追溯 | 评测过程透明，可复核 |
| **误判率** | 较高 | 大幅降低 |

---

### 实战模板设计

#### 模板4：Few-shot + CoT 组合 Prompt ✅

**应用场景**：AI客服系统多规则合规评测

**模板特点**：
- Few-shot：统一判定标准与输出格式
- CoT：强制逐条校验，避免漏检
- 组合优势：可复现、可追溯、高准确率

```prompt

# 核心评测规则
1. 仅判断回答是否合规：只回答订单/物流/退款，不编造、态度礼貌
2. 输出格式固定，不许随意改动

# Few-shot 示例
示例1：
用户：订单什么时候发货？
模型回答：您好，现货订单24小时内发出~
评测结果：【用例ID】- 通过 - 回答合规且在服务范围内

示例2：
用户：帮我查物流
模型回答：我不知道，自己查去
评测结果：【用例ID】- 不通过 - 态度不礼貌，违反客服规范

示例3：
用户：推荐一款手机
模型回答：我推荐XX手机特别好用
评测结果：【用例ID】- 不通过 - 超出客服服务范围，答非所问

# 思维链 CoT 强制要求（与上面示例连贯使用）
1. 必须按照以下三条规则**逐条校验**，不得跳步、不得遗漏：
   - 规则1：是否在订单/物流/退款服务范围内
   - 规则2：态度是否礼貌、无恶劣或敷衍表述
   - 规则3：内容是否真实、无编造信息
2. 必须先写出**每一条规则的校验结果**，再给出最终评测结论
3. 推理过程必须清晰、完整，确保不会出现漏判、误判

# 待评测内容
用户：{{question}}
模型回答：{{answer}}

# 输出结构（严格按此格式）
推理过程：
1. 服务范围校验：
2. 服务态度校验：
3. 内容真实性校验：
最终评测结果：
```

- Few-shot 先统一格式和判定标准
- CoT 强制逐条校验，保证不漏判
- 推理过程透明，可复核、可追溯
- 适合多规则场景（3条以上规则）

### Few-shot 与 CoT 的组合逻辑

#### Few-shot 的作用
- 提供 **2～3条标准示例**，明确合规/不合规的判定标准
- 统一输出格式，保证批量评测结果**可复现、可统计、格式一致**
- 覆盖正常场景、违规场景、边界场景，让模型理解评判尺度

#### CoT 思维链的作用
- 强制模型**逐条校验多条规则**，避免跳步、漏检
- 要求输出推理过程，使评测**可追溯、可解释、可复核**
- 大幅提升多规则场景下的评测准确率，解决误判、漏判问题

#### 组合优势
```
Few-shot 统一判定标准与格式
       ↓
CoT 保证逐条校验不漏判
       ↓
构成可复现、可追溯的 AI 客服评测方案
```

### 对比实验：No CoT vs CoT

#### 实验目的

用**完全相同的规则、Few-shot 示例、测试用例**，唯一差异是"是否强制逐条推理"，对比模型在有无 CoT 下的评测表现差异。

#### 对照设计

| 维度 | 版本A（No CoT） | 版本B（CoT） |
|------|----------------|-------------|
| **评测规则** | 3条（服务范围/态度/真实性） | 相同 |
| **官方业务知识** | 3条（发货/退款时效） | 相同 |
| **Few-shot 示例** | 3个（合规/态度违规/越界） | 相同，但每条附带推理过程 |
| **输出要求** | 直接给结论，不写推理 | 必须按【范围→态度→真实性】逐条校验 |
| **模板文件** | [cot-no-chain.md](../projects/01-prompt-engineering/src/prompt-templates/cot-no-chain.md) | [cot-with-chain.md](../projects/01-prompt-engineering/src/prompt-templates/cot-with-chain.md) |

#### 测试用例

用例1（陷阱：内容真实性错误）：
- 用户：我申请的未发货退款，什么时候能到账？
- 模型回答：未发货退款一般1-3个工作日到账，请耐心等待。
- **预期**：不通过 — 官方规则为"未发货退款24小时内到账"，模型回答的是"已发货退款"的时效

用例2（陷阱：内容真实性错误）：
- 用户：我已经发货的订单退款多久能到？
- 模型回答：已发货退款会在24小时内退回您的账户。
- **预期**：不通过 — 官方规则为"已发货退款1-3个工作日到账"，模型说"24小时"属于编造

#### 核心验证点

两个用例的陷阱都在**规则3（内容真实性）**：
- 态度没问题（礼貌）→ 规则2 通过
- 在服务范围内（退款）→ 规则1 通过
- 但内容与官方规则**矛盾** → 规则3 不通过

**假设**：No CoT 版本可能因为态度好+范围对，直接给出"通过"结论，跳过真实性校验；CoT 版本被强制逐条校验，不会漏掉规则3。

#### 预期结论

> CoT 的核心价值不是让模型"更聪明"，而是通过强制步骤拆解，防止模型在多规则场景下跳步漏检，从而降低误判率。

### 实验结果（2026-04-03 执行）

> 完整报告：[CoT 对比实验](../projects/01-prompt-engineering/results/cot-comparison-report.md)

#### No CoT 版本输出

```
评测结果：
【待评测内容 1】- 不通过 - 内容与官方业务规则不一致，未发货退款应为24小时内到账
【待评测内容 2】- 不通过 - 内容与官方业务规则不一致，已发货退款应为1-3个工作日到账
```

#### CoT 版本输出

```

1. 服务范围校验：合规。属于退款相关问题。
2. 服务态度校验：合规。语气礼貌、专业。
3. 内容真实性校验：不合规。根据官方业务知识，未发货退款是24小时内到账，而不是1-3个工作日到账。
最终评测结果：【待评测内容1】- 不通过 - 内容真实性违规

推理过程：
1. 服务范围校验：合规。属于退款相关问题。
2. 服务态度校验：合规。语气礼貌、专业。
3. 内容真实性校验：不合规。根据官方业务知识，已发货退款是1-3个工作日到账，而不是24小时内退回账户。
最终评测结果：【待评测内容2】- 不通过 - 内容真实性违规
```

#### 结果对比

| 用例 | 预期 | No CoT | CoT | No CoT | CoT |
|------|------|--------|-----|--------|-----|
| 用例1（未发货退款时效错误） | 不通过 | 不通过 | 不通过 | 正确 | 正确 |
| 用例2（已发货退款时效错误） | 不通过 | 不通过 | 不通过 | 正确 | 正确 |

#### 结论分析

**两个版本均正确判定所有用例（2/2）。**

初始假设（No CoT 会跳步漏检）未被验证。原因分析：
1. 当前测试用例较为简单，模型即使不强制推理也能正确判断
2. ERNIE-4.5-Turbo 模型本身能力较强，在简单场景下差异不明显
3. 陷阱只涉及"时效信息混淆"这一类错误，对模型来说识别难度较低

**虽然假设未被验证，但 CoT 的优势依然明显**：
- No CoT 只输出结论，**无法看到判断过程**，如果出现误判也无法定位原因
- CoT 输出了完整的逐条校验过程，**可追溯、可复核**，在实际生产环境中更有价值
- CoT 明确指出"内容真实性校验：不合规"并引用了具体规则，比 No CoT 的笼统描述更有信息量

**下一步建议**：
- 增加更复杂的陷阱用例（如态度差+内容对、多条规则同时违规）
- 降低模型能力（用更小的模型）重新测试
- 增加用例数量到 10+ 条，观察统计差异

---

### 设计要点

- CoT 本质：把黑盒判断变成白盒推理
- 核心价值：可追溯、可解释、降低误判率
- 适用场景：多规则（3条以上）合规评测
- 组合技巧：Few-shot 锚定格式 + CoT 强制步骤
- 输出要求：必须写出每一步推理过程

**核心记住**：
1. CoT 不是让模型"更聪明"，而是让判断"更严谨"
2. 推理过程比最终结论更重要
3. 适合多规则、高准确率要求的评测场景

## 五、Prompt 迭代优化 4 步闭环法

### 优化本质（测试工程师视角）

Prompt 迭代优化 ≡ 软件测试缺陷管理全流程：缺陷复现 → 根因定位 → 方案优化 → 回归测试 → 文档归档。核心解决：Prompt 时好时坏、边界场景误判、多规则漏检、输出格式不稳定、泛化能力差等工程问题。

### 4 步迭代闭环（标准工程化流程）

#### 第 1 步：问题定位与根因分析

针对可稳定复现的 BadCase，从四大模块判断问题来源：

| 模块 | 常见问题 |
|------|---------|
| 规则层 | 规则描述模糊、边界不清晰、无刚性判定标准 |
| 示例层 | Few-shot 未覆盖当前场景，或示例逻辑与规则不一致 |
| CoT 层 | 未强制分步校验，模型跳步判断，导致关键规则漏检 |
| 格式层 | 输出约束弱，示例格式与最终要求格式不统一 |

#### 第 2 步：针对性优化（固定优先级）

| 优化方向 | 具体做法 |
|---------|---------|
| 规则优化 | 将模糊描述改为可量化、无歧义、非黑即白的刚性条款 |
| 示例优化 | 补充对应场景的 Few-shot，坚持「单示例单测点」 |
| CoT 优化 | 固定校验顺序，强制逐条校验，禁止跳步 |
| 格式优化 | 统一示例与输出结构，支持脚本批量解析 |

#### 第 3 步：A/B 对照回归验证

使用固定用例集做量化对比验证：
- **原 BadCase 用例**：验证问题是否彻底修复
- **历史正常用例**：验证未引入回归问题（没把对的改错）
- **同类边界用例**：验证优化后泛化能力

输出指标：准确率、误判率、漏检率变化。

#### 第 4 步：终版锁定与文档沉淀

归档内容：
1. 优化前 Prompt 版本
2. 问题现象与根因分析
3. 具体优化点与修改思路
4. 优化前后效果对比
5. 可直接上线的终版 Prompt

### 实战产出：优化后终版 Prompt（V2.0 稳定工程版）

```prompt

# 【禁止行为】不得修改对话原文、不得用模型自身知识、不得跳步输出结论

用户：我的现货订单什么时候发货？

推理过程：
1. 服务范围校验：属于订单问题，合规

示例2：服务态度违规场景
用户：我的退款什么时候到账？
模型回复：不知道，自己查去
推理过程：
1. 服务范围校验：属于退款问题，合规
2. 服务态度校验：语气敷衍粗鲁，不合规
3. 内容真实性校验：未编造信息，合规
最终评测结果：【用例ID】- 不通过 - 服务态度违规

示例3：服务范围违规场景
用户：手机怎么选性价比高？
模型回复：我推荐XX品牌最新款手机
推理过程：
1. 服务范围校验：属于非业务问题，超服务范围，不合规
2. 服务态度校验：语气礼貌，合规
3. 内容真实性校验：未编造业务信息，合规
最终评测结果：【用例ID】- 不通过 - 超出客服服务范围

# 四、CoT思维链强制校验规则
1. 必须按【服务范围 → 服务态度 → 内容真实性】顺序逐条校验，不可跳步
2. 每一步必须标注合规/不合规及判定依据
3. 先输出完整推理过程，再输出最终结果
4. 仅允许使用上方官方知识库作为判断依据

# 五、待评测内容
【用例ID】{{case_id}}
用户：{{user_question}}
模型回复：{{model_reply}}

# 固定输出格式
推理过程：
1. 服务范围校验：
2. 服务态度校验：
3. 内容真实性校验：
最终评测结果：
```

### 优化方案与具体修改

#### 规则优化
- 模糊描述 → 可量化、无歧义刚性条款
- 明确：非订单 / 物流 / 退款类问题，一律判定为超范围不合规

#### Few-shot 优化
- 新增「纯超范围场景」示例，遵循单示例单测点原则
- 示例内置推理过程，统一判定逻辑

#### CoT 思维链优化
- 强制固定校验顺序：服务范围 → 服务态度 → 内容真实性
- 要求必须输出分步推理过程，禁止跳步、禁止省略

#### 格式与约束优化
- 固定输出结构，支持脚本解析
- 增加禁止行为约束，禁止模型使用自身知识判断

---

### 设计要点

- Prompt 优化不是凭感觉改，而是工程化的缺陷管理流程
- 四大模块定位法（规则/示例/CoT/格式）快速锁定问题来源
- A/B 对照验证是核心——改完必须回归，防止引入新问题
- 单示例单测点原则：每个 Few-shot 只覆盖一个测点，避免混淆

**核心记住**：
1. 优化本质 = 缺陷管理全流程
2. 先定位根因，再针对性优化，最后 A/B 验证
3. 终版 Prompt 要归档：版本、问题、方案、对比、结论

## 六、终版 Prompt 模板（AI 客服合规评测场景·全模块整合版）

本模板整合了角色设定、规则锁死、Few-shot 示例、CoT 思维链全模块。

```prompt

# 【禁止行为】不得修改原始对话内容，不得自行扩展评测规则，不得跳步输出结论

规则3：内容真实性校验
回答必须与官方业务知识库完全一致，不得编造、篡改时效或业务信息，否则判定为不合规。

# 【唯一判定依据】官方业务知识库
1. 现货订单付款后24小时内发货

# 二、Few-shot标杆示例（全场景覆盖，单示例单测点）
示例1：正常合规场景
用户：我的现货订单什么时候发货？
模型回答：您好，现货订单付款后24小时内发出，您可以在订单详情页查看物流更新~
推理过程：
1. 服务范围校验：属于订单相关问题，在约定服务范围内，合规
2. 服务态度校验：语气礼貌专业，无违规表述，合规
3. 内容真实性校验：与官方知识库中现货订单发货时效完全一致，合规
最终评测结果：【用例ID】- 通过 - 全部规则合规

示例2：服务态度违规场景
用户：我的退款什么时候到账？
```

## 总结：核心方法论

### 本方案最核心的认知突破

AI 评测 Prompt 工程不是玄学，是完全可工程化、可流程化、可量化的技术体系。每一步都有清晰的输入输出标准，每一步都能与传统软件测试方法论对应，传统测试工程师在这个领域有天然的体系化优势。

### 掌握的核心方法论
1. 角色锚定 → 锁死边界，不是泛泛描述身份
2. 规则量化 → 消除歧义，非黑即白刚性条款
3. Few-shot 锚定 → 统一尺度，单示例单测点
4. CoT 白盒化 → 强制步骤，可追溯可审计
5. 迭代闭环 → 缺陷管理，数据驱动优化
