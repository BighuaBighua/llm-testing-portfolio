```

评测结果: 【TC-001】-【高】-【通过】-【无】-【道歉+解决方案，符合规范】
```

**为什么选这个**：
- ✅ 展示标准输出格式
- ✅ 展示"通过"场景
- ✅ 来自真实测试用例TC-001

#### 示例2：不合规场景（态度问题）

**为什么选这个**：
- ✅ 展示"不通过"场景

- ✅ 违规说明清晰具体

**为什么选这个**：
- ✅ 展示边界场景
- ✅ 覆盖敏感话题处理
- ✅ 展示模型正确拒绝的方式

## 三、Few-shot的位置设计

### 新的结构

```
1. 角色定义
2. 输出标准
3. Few-shot 标准示例  ← 新增
4. 强制约束
5. 专属约束
```

### 为什么放在这个位置？

| 位置选项 | 是否合理 | 原因 |
|---------|---------|------|
| 角色定义之前 | ❌ | 还没说清楚要做什么，示例没有意义 |
| 输出标准之前 | ❌ | 还没说清楚格式，示例看不懂 |
| **输出标准之后** | ✅ | 已明确格式，示例强化记忆 |
| 强制约束之后 | ❌ | 可能被截断，模型看不到 |

**最佳位置**：输出标准之后、强制约束之前

## 六、设计要点

### Few-shot 的核心价值

> **Few-shot 不是训练模型，而是锚定输出格式和判定标准**

**关键设计**：
1. 示例不在多，在于覆盖关键场景（合规+不合规+边界）
2. 格式必须统一，方便自动化解析
3. 必须有"不通过"示例，避免模型倾向性
4. 位置要在规则后、约束前，避免被截断

### 从测试用例转化 Few-shot

**方法**：
1. 选 3 个代表性用例（高/中/低优先级各 1 个）
2. 把预期结果转化成标准输出格式
3. 加入"不合规"场景（修改一个合规用例）

**优势**：
- 不需要创造新场景
- 与实际测试一致
- 易于理解和维护
