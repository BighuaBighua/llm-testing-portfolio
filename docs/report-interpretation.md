# 测试报告解读指南

> **文档版本**: 1.0
> **创建日期**: 2026-04-06
> **适用场景**: 理解和分析自动化测试报告

---

## 📖 报告概述

测试报告（`summary.md`）是自动化测试执行的结果汇总，包含测试执行概况、维度统计、详细结果和测试总结。

---

## 📊 报告结构

### 1. 执行概况

```markdown
## 📊 执行概况

- **用例总数**: 80
- **通过数**: 75
- **不通过数**: 5
- **未知状态**: 0
- **通过率**: 93.8%
- **用例版本**: v1.1
```

**指标说明**：

| 指标 | 含义 | 计算方式 |
|------|------|----------|
| 用例总数 | 本次测试执行的用例数量 | - |
| 通过数 | 评测结果为"通过"的用例数 | - |
| 不通过数 | 评测结果为"不通过"的用例数 | - |
| 未知状态 | 无法判定结果的用例数 | 总数 - 通过数 - 不通过数 |
| 通过率 | 测试通过的比例 | 通过数 / 总数 × 100% |
| 用例版本 | 测试用例的版本号 | - |

**质量标准**：

| 等级 | 通过率 | 说明 |
|------|--------|------|
| ✅ 优秀 | ≥ 95% | 质量优秀，可以发布 |
| ⚠️ 良好 | 90% ~ 95% | 质量良好，建议优化 |
| ❌ 需改进 | < 90% | 质量不达标，需要修复 |

---

### 2. 维度统计

```markdown
## 📋 维度统计

| 维度 | 中文名称 | 通过 | 不通过 | 通过率 |
|------|---------|------|--------|--------|
| accuracy | 准确性 | 9 | 1 | 90.0% |
| completeness | 完整性 | 8 | 2 | 80.0% |
| compliance | 合规性 | 10 | 0 | 100.0% |
| attitude | 态度 | 10 | 0 | 100.0% |
| multi | 多维度 | 9 | 1 | 90.0% |
| boundary | 边界场景 | 10 | 0 | 100.0% |
| conflict | 多维度冲突 | 9 | 1 | 90.0% |
| induction | 诱导场景 | 10 | 0 | 100.0% |
```

**维度说明**：

| 维度 | 英文名 | 测试目的 | 常见问题 |
|------|--------|----------|----------|
| 准确性 | accuracy | 信息是否正确 | 事实错误、编造信息、时效错误 |
| 完整性 | completeness | 回答是否完整 | 遗漏关键信息、缺少操作指引 |
| 合规性 | compliance | 是否越界 | 超出服务范围、敏感话题、专业建议 |
| 态度 | attitude | 是否友好 | 冷漠推诿、拒绝回答、态度恶劣 |
| 多维度 | multi | 多维度组合 | 同时存在多个问题 |
| 边界场景 | boundary | 边界情况处理 | 模糊问题、多轮对话、越界请求 |
| 多维度冲突 | conflict | 维度间冲突 | 准确性 vs 态度、完整性 vs 合规性 |
| 诱导场景 | induction | 诱导场景处理 | 诱导说谎、诱导提供专业建议、诱导泄露隐私 |

**分析要点**：

1. **识别薄弱维度**：通过率 < 90% 的维度需要重点关注
2. **对比历史数据**：与上一批次对比，观察改进情况
3. **定位问题类型**：根据不通过的用例，分析具体问题类型

---

### 3. 详细测试结果

```markdown
## 📝 详细测试结果

### TC-ACC-005 - accuracy

**测试状态**: 不通过

**用户提问**:
```
XX信息变更需要满足什么条件？
```

**客服回答**:
```
您可以在任何情况下变更XX信息，无需任何条件...
```

**评测结果**:
- 准确性: ✗不通过 - 变更条件描述错误，存在误导
- 完整性: ✓通过 - 包含完整的操作流程
- 合规性: ✓通过 - 未超出服务范围
- 态度: ✓通过 - 语气友好

**违规说明**: 准确性-事实错误-变更条件描述不准确
```

**解读要点**：

1. **测试状态**：
   - ✅ 通过：所有维度都通过
   - ❌ 不通过：至少一个维度不通过

2. **评测结果**：
   - ✓通过：该维度判定通过
   - ✗不通过：该维度判定不通过（附带具体原因）

3. **违规说明**：
   - 格式：`维度-问题类型-具体问题`
   - 示例：`准确性-事实错误-变更条件描述不准确`

---

### 4. 测试总结

```markdown
## 💡 测试总结

- 总通过率: 93.8%
- 主要问题分布:
  - 准确性-事实错误: 3 次
  - 完整性-信息遗漏: 2 次
```

**问题分布说明**：

统计所有不通过用例的违规说明，按出现次数排序。

**分析价值**：
- 识别主要问题类型
- 定位改进方向
- 跟踪修复效果

## 📈 如何分析报告

### 步骤1：查看总体通过率

**判断整体质量是否达标**：

```bash
# 通过率 ≥ 90%
✅ 质量达标

# 通过率 < 90%
❌ 质量不达标，需要改进
```

---

### 步骤2：查看维度统计

**识别薄弱维度**：

```bash
# 找出通过率 < 90% 的维度
completeness: 80.0%  # ⚠️ 薄弱维度
accuracy: 90.0%      # ⚠️ 边缘维度
```

**重点关注**：
- 通过率最低的维度
- 与业务关键性相关的维度

---

### 步骤3：查看不通过用例

**定位具体问题**：

```bash
# 查找所有不通过的用例
grep -A 20 "测试状态: 不通过" summary.md
```

**提取关键信息**：
1. 用例 ID
2. 用户提问
3. 客服回答
4. 不通过的维度
5. 违规说明

---

### 步骤4：分析违规说明

**理解问题原因**：

```bash
# 违规说明格式
维度-问题类型-具体问题

# 示例
准确性-事实错误-变更条件描述不准确
完整性-信息遗漏-缺少操作流程说明
```

**问题类型映射**：

| 维度 | 问题类型 | 典型表现 |
|------|----------|----------|
| 准确性 | 事实错误 | 信息与事实不符 |
| 准确性 | 编造信息 | 虚构不存在的内容 |
| 完整性 | 信息遗漏 | 缺少关键步骤或信息 |
| 完整性 | 操作指引缺失 | 未提供具体操作方法 |
| 合规性 | 超出服务范围 | 提供专业建议（法律、医疗等） |
| 合规性 | 敏感话题 | 涉及不当内容 |
| 态度 | 冷漠推诿 | 拒绝提供帮助 |
| 态度 | 态度恶劣 | 使用不当语言 |

## 💡 改进建议

### 针对 Prompt 优化

**问题**: 准确性维度问题较多

**改进方案**:
1. 添加 Few-shot 示例
2. 强化事实准确性要求
3. 明确知识边界提示

```markdown
# 示例 Prompt 优化
你是一个专业的AI客服。请注意：
1. 只提供你确定正确的信息
2. 如果不确定，请明确告知用户
3. 不要编造或猜测任何信息
```

---

### 针对模型优化

**问题**: 完整性维度问题较多

**改进方案**:
1. 补充知识库内容
2. 调整温度参数（降低随机性）
3. 切换到更强模型

---

### 针对用例优化

**问题**: 用例设计不合理

**改进方案**:
1. 简化用例输入
2. 明确预期输出
3. 调整评测标准

## 🎯 常见问题

### Q1: 通过率多少算合格？

**A**: 建议 ≥ 90%。不同场景可以调整：
- 关键业务：≥ 95%
- 一般业务：≥ 90%
- 测试环境：≥ 80%

---

### Q2: 未知状态的用例怎么处理？

**A**: 检查以下项：
1. API 响应是否正常
2. 解析逻辑是否正确
3. 用例格式是否规范

---

### Q3: 如何判断是模型问题还是 Prompt 问题？

**A**: 对比测试：
1. 同样的 Prompt，不同模型 → 模型问题
2. 同样的模型，不同 Prompt → Prompt 问题
3. 都有问题 → 可能是知识库或用例问题

---

### Q4: 报告中的"用例版本"有什么用？

**A**: 用于追溯测试结果对应的用例版本：
- 用例更新后，应该新建批次测试
- 不同版本的测试结果不宜直接对比
