# 中断恢复用户指南

## 功能介绍

测试执行过程中可能因网络故障、API 限流、程序崩溃等原因中断。
中断恢复功能可以在不重复执行已完成用例的情况下，继续执行未完成的用例。

## 使用场景

1. API 限流导致中断
2. 网络故障导致中断
3. 程序崩溃导致中断
4. 主动中断后恢复

## 操作步骤

### 步骤1：检测中断情况

```bash
python3 scripts/run_tests.py --report-only --batch-id batch-004
```

### 步骤2：查看执行日志

```bash
cat projects/01-ai-customer-service/results/batch-004/test_execution.log
```

### 步骤3：恢复执行

```bash
python3 scripts/run_tests.py --mode incremental --report append --batch-id batch-004
```

## 注意事项

1. 环境一致性：恢复前确认模型、API 未变更
2. 用例版本：确认用例文件未修改
3. 网络稳定：确保网络连接稳定

## 常见问题

Q: 如何判断是否需要恢复？
A: 检查 test_execution.log 中的完成进度。

Q: 恢复后会重复执行吗？
A: 不会，incremental 模式会跳过已完成的用例。

**文档大纲**：

```markdown
# 测试报告解读指南

## 报告结构

测试报告（summary.md）包含以下部分：

### 1. 执行概况

```markdown
- 用例总数: 80
- 通过数: 75
- 不通过数: 5
- 通过率: 93.8%
```

**解读**：
- 总体质量良好（通过率 > 90%）
- 需要关注 5 条不通过的用例

### 2. 维度统计

| 维度 | 通过 | 不通过 | 通过率 |
|------|------|--------|--------|
| accuracy | 9 | 1 | 90% |
| completeness | 8 | 2 | 80% |

**解读**：
- completeness 维度问题较多
- 需要重点优化完整性

### 3. 详细测试结果

每个用例的详细评测结果：

- 用户提问
- 客服回答
- 评测结果（各维度判定）
- 违规说明

## 指标含义

### 通过率

通过率 = 通过数 / 总数 × 100%

**质量标准**：
- ✅ 优秀: ≥ 95%
- ⚠️ 良好: 90% ~ 95%
- ❌ 需改进: < 90%

### 维度得分

各维度的通过率反映不同方面的质量：

- **准确性**: 信息是否正确
- **完整性**: 回答是否完整
- **合规性**: 是否越界
- **态度**: 是否友好

## 如何分析报告

### 步骤1：查看总体通过率

判断整体质量是否达标。

### 步骤2：查看维度统计

识别薄弱维度。

### 步骤3：查看不通过用例

定位具体问题。

### 步骤4：分析违规说明

理解问题原因。

## 问题定位

### 问题1：准确性不通过

**表现**: 信息错误、编造内容

**定位方法**:
1. 检查用例输入是否明确
2. 检查知识库是否有答案
3. 检查模型是否产生幻觉

### 问题2：完整性不通过

**表现**: 缺少关键信息

**定位方法**:
1. 检查用例是否要求完整回答
2. 检查模型是否理解要求
3. 优化 Prompt 提示完整性要求
```

**文档大纲**：

```markdown
# Bad Case 分析文档

## 什么是 Bad Case

Bad Case 是指测试不通过的用例，代表模型表现不佳的场景。

## 收集方法

### 方法1：从测试报告中提取

```bash
# 提取所有不通过的用例
grep -A 20 "不通过" results/batch-004/summary.md
```

### 方法2：从 results.json 提取

```python
import json

with open('results.json', 'r') as f:
    results = json.load(f)

bad_cases = [r for r in results if r['evaluation_result']['status'] == '不通过']
```

## 分析框架

### 维度分析

按维度统计 Bad Case 分布：

| 维度 | Bad Case 数量 | 占比 |
|------|--------------|------|
| accuracy | 2 | 40% |
| completeness | 3 | 60% |

### 问题类型分析

按问题类型分类：

| 问题类型 | 示例 | 数量 |
|---------|------|------|
| 事实错误 | 编造信息 | 2 |
| 信息遗漏 | 缺少关键步骤 | 3 |

### 根因分析

分析问题根本原因：

1. **Prompt 设计问题**
   - 提示不够明确
   - 缺少 Few-shot 示例

2. **模型能力问题**
   - 知识库缺失
   - 推理能力不足

3. **用例设计问题**
   - 用例过于复杂
   - 输入不明确

## 改进建议

### 针对 Prompt 优化

1. 添加 Few-shot 示例
2. 明确输出格式要求
3. 强化关键规则

### 针对模型优化

1. 补充知识库
2. 调整温度参数
3. 切换更强模型

### 针对用例优化

1. 简化用例输入
2. 明确预期输出
3. 调整评测标准
```
